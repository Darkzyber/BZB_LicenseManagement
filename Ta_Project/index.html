🧩 Proposed Modular Architecture — TA Domain

แต่ละ module = 1 HTTP-triggered child flow รับ JSON → ทำงาน → return JSON กลับ

🔵 moduleGetEmployee

Get employee record จาก SharePoint โดย filter จาก EmployeeID หรือ Email — return object ครบ

Input: filterType, filterValue
Output: employeeObj (FullName, Dept, Manager, etc.)

📌 ใช้โดย: Probation, NewJoiner, Contract, Resign, Transfer (8 flows)

🔵 moduleUpsertSPItem

Upsert (create/update) item ใน SharePoint list ที่กำหนด โดยรับ filterField + fields object

Input: siteUrl, listName, filterField, filterValue, fields
Output: action, itemId

📌 ใช้โดย: ทุก process flow (15+ flows)

🔵 moduleUpdateStatus

Update เฉพาะ Status / ProcessingStatus field ใน SP item — เพื่อลด permission risk

Input: siteUrl, listName, itemId, status, processingStatus
Output: itemId

📌 ใช้โดย: ทุก approval flow (15+ flows)

🟡 moduleApprovalEngine

จัดการ multi-tier approval ผ่าน ApprovalRegistry list — รับ chain JSON แล้วรัน step ต่อ step

Input: requestID, requestType, chain JSON, sourceFlowID
Output: status, currentStep, rejectedReason

📌 ใช้โดย: OrgChart, Hiring, Probation, Contract, Transfer (11 flows)

🟡 moduleApproverChainBuilder

สร้าง approver chain โดยใช้ Claude API หรือ rule table — ส่งคืน JSON chain พร้อมใช้

Input: requestType, dept, grade, employeeID
Output: totalSteps, chain[]

📌 ใช้โดย: ทุก flow ที่มี dynamic approver

🟢 moduleNotifyEmail

ส่ง email notification ไปหา requester/approver พร้อม HTML template ตาม requestType

Input: toEmail, requestType, status, requestID, rejectedReason?
Output: messageId

📌 ใช้โดย: 9 flows ทุก domain

🟢 moduleNotifyTeams

ส่ง adaptive card ไปยัง Teams channel ที่กำหนด พร้อม action button

Input: channelType, title, message, requestID, actionURL?
Output: messageId

📌 ใช้โดย: OrgChart, Hiring, Probation, Error alerts (7 flows)

🟣 moduleWriteLog

เขียน audit log entry ลง Logs list พร้อม timestamp, actor, flowID, action, remark

Input: requestType, requestID, action, actor, flowID, remark?
Output: logItemId

📌 ใช้โดย: ทุก flow ที่ต้องการ audit trail (10+ flows)

🟣 moduleLogError

Catch error → Create Ticket ใน error list → Teams alert (มีอยู่แล้ว — ต้อง standardize)

Input: flowID, errorMessage, requestID, context
Output: ticketID

📌 ใช้โดย: 37 flows ✅ มีแล้ว ต้อง centralize

🟠 moduleGenerateDOCX

สร้าง DOCX จาก Muhimbi template โดยรับ templateName + fields object — save to SP + email

Input: templateName, fields, saveToSP, emailTo?
Output: fileUrl, emailSent

📌 ใช้โดย: OfferLetter, Contract, AutogenDOCX (3 flows)

Process Flow Architecture (หลัง Modularization)

ทุก Process Flow ใช้โครงสร้าง 5 ส่วน:

1. TRIGGER
   └─ SharePoint form / HTTP / Schedule

2. PRE-PROCESS (HTTP POST → module)
   ├─ moduleGetEmployee        → ดึงข้อมูล employee
   ├─ moduleApproverChainBuilder → สร้าง chain
   └─ moduleUpsertSPItem       → บันทึก ProcessList

3. APPROVAL ENGINE (HTTP POST → moduleApprovalEngine)
   ├─ Create ApprovalRegistry record
   ├─ Run Step 1..N (sequential)
   │   ├─ Approved → Next step
   │   │   └─ Complete → moduleNotifyEmail + moduleWriteLog
   │   └─ Rejected → moduleNotifyEmail + moduleWriteLog
   └─ Update ProcessList.ApprovalComplete

4. POST-PROCESS (HTTP POST → module)
   ├─ moduleUpdateStatus        → update status
   ├─ moduleGenerateDOCX        → สร้างเอกสาร (ถ้ามี)
   └─ moduleUpsertSPItem        → update related lists

5. NOTIFY
   ├─ moduleNotifyEmail         → แจ้ง requester
   └─ moduleNotifyTeams         → แจ้ง HR channel
