System Architecture
❓
ทำไมพนักงานต้อง Request ก่อน? ทำไมไม่ให้ HR/IT Admin สร้าง License เองได้เลย?
Design DecisionWhy
▾

เพราะต้องการ audit trail และ approval gate ก่อนเสียเงิน ถ้า IT Admin สร้างเองได้เลย จะไม่มีหลักฐานว่าใครขอ ทำไมถึงต้องการ และ Manager อนุมัติหรือเปล่า

Flow ที่ถูกต้องคือ: Employee Request → Manager Approve → Purchase → License Assigned ทุกขั้นตอนมีคนรับผิดชอบชัดเจน

📌 License Management (Phase 3) จะ unlock ได้ก็ต่อเมื่อผ่าน Approval และ Purchase Write-Back แล้วเท่านั้น
❓
สรุป 3 Phases ของระบบคืออะไร?
How It Works
▾

Phase 1 — Request & Approval: พนักงานกรอก Form ใน Power Apps → Flow ส่ง Approval ให้ Manager → Manager Approve/Reject → ถ้า Approve ระบุ Payment Method (PR/PO หรือ Corporate Card)

Phase 2 — Purchase + Write-Back: จัดซื้อ/การเงินดำเนินการตามช่องทาง → เสร็จแล้วต้อง Write-Back เลข Reference (PO Number หรือ CC Transaction ID) กลับมาใน SharePoint → set PurchaseStatus = "Completed"

Phase 3 — License Management: Flow 2 detect Completed → แจ้ง IT Admin → IT Admin กรอก License Key + Assign ให้พนักงาน → ระบบเริ่ม Monitor Expiry รายวัน

🔑 Key concept: PurchaseStatus = "Completed" คือประตูที่เปิด Phase 3 — ไม่มี Write-Back = ไม่มี License
❓
แต่ละ Stakeholder มีบทบาทอะไรในระบบ?
How It Works
▾

พนักงาน ยื่น Request ผ่าน Power Apps รับแจ้งผลอนุมัติ/ปฏิเสธ รับแจ้งเมื่อ License พร้อม

Manager Approve/Reject Request ตัดสินใจ Renew/Cancel เมื่อ License ใกล้หมด

HR/IT Admin Write-Back Reference Number, Assign License Key, ดำเนินการ Movement เมื่อพนักงาน Resign/Transfer, ลงมือ Cancel/Renew ที่ Vendor Portal จริงๆ

จัดซื้อ ออก PO ติดต่อ Vendor — process ของเขาเอง ระบบเราแค่รอเลข PO กลับมา

การเงิน ชำระ Corporate Card — process ของเขาเอง ระบบเราแค่รอ CC Transaction ID กลับมา

💰
Purchase Flow — PR/PO & Corporate Card
❓
ระบบเราต้องไปยุ่งกับ Process ของจัดซื้อหรือการเงินไหม?
Design Decision
▾

ไม่ต้องยุ่งเลย Process ข้างในของจัดซื้อ (ออก PO, ติดต่อ Vendor) และการเงิน (อนุมัติ Corporate Card) เป็นหน้าที่ของเขา ระบบเราแค่:

✅ สิ่งที่ระบบทำ
ส่ง Notification ว่า "Request ผ่านแล้ว ช่องทางนี้"
รอให้คนกรอก Reference กลับมา
Detect เมื่อ PurchaseStatus = "Completed"
❌ สิ่งที่ไม่ใช่หน้าที่เรา
Process ออก PO ของจัดซื้อ
Process อนุมัติ Corporate Card
จัดการ Vendor ต่างๆ
❓
Write-Back คืออะไร ใครกรอก?
How It Works
▾

Write-Back คือการที่ HR/IT Admin กรอก Reference Number ของการจัดซื้อกลับเข้าไปใน SharePoint หลังจากได้รับข้อมูลจากจัดซื้อ/การเงิน

PR/PO path: จัดซื้อแจ้ง PO Number มาให้ → HR/IT Admin กรอก PONumber ใน Power Apps Admin Screen → set PurchaseStatus = "Completed"

Corporate Card path: การเงินแจ้ง Transaction ID มาให้ → HR/IT Admin กรอก CCRefID ใน Power Apps → set PurchaseStatus = "Completed"

⚠️ ใครกรอก Write-Back? HR/IT Admin เป็นคนกรอก ไม่ใช่จัดซื้อหรือการเงินเข้ามากรอกเอง (ไม่คุ้มสร้าง App แยกสำหรับ 1 field)
🔁
License Lifecycle — Movement & Expiry
❓
เมื่อพนักงาน Resign หรือ Transfer แผนก ต้องทำอะไรกับ License?
Priority 0หลุดลอย
▾

IT Admin ต้องดำเนินการ ก่อนปิด AD Account พนักงาน (ถ้าปิด Account ก่อน License จะหลุดลอยทันที หาเจ้าของไม่ได้)

มี 3 Actions ให้เลือก:

Transfer → โอน License ให้พนักงานคนอื่น ผล: AssignedTo เปลี่ยน ใช้งานต่อได้ทันที ไม่ต้องซื้อใหม่
Return to Pool → คืน License เข้าระบบ ยังไม่ Assign ใคร รอ Request ครั้งต่อไป
Cancel → ยกเลิก License ไม่ต้องการแล้ว IT Admin ต้องไป Cancel ที่ Vendor Portal ด้วย
⚠️ ลำดับที่ถูกต้อง: HR แจ้ง IT Admin → IT Admin ทำ License Movement ใน Power Apps → Flow 4 อัปเดต SharePoint + ส่ง Email → จากนั้นค่อยปิด AD Account
❓
License หมดอายุ — ระบบแค่แจ้งเตือนพอไหม?
Priority 0Design Decision
▾

ไม่พอ การแค่ "ส่ง Email แจ้งเตือน" คือ pattern ที่ผิด เพราะไม่มีคนรับผิดชอบจนถึงจุดจบ

ระบบต้องบังคับให้มีคน ตัดสินใจ (Renew/Cancel/Transfer) และ ลงมือ ที่ Vendor Portal จริงๆ:

Timeline: 30 วัน → แจ้ง + รอตัดสินใจ   7 วัน → Escalate ถ้าไม่มี Action   1 วัน → แจ้ง IT Admin ด่วน   หมดแล้ว → ส่ง Task ให้ IT Admin ยืนยัน

ถ้าไม่ Cancel ที่ Vendor Portal = อาจถูก Auto-renew และเสียเงินทั้งที่ไม่ต้องการแล้ว
❓
ใครมีอำนาจเข้า Vendor Portal เพื่อ Cancel/Renew Subscription?
Design Decision
▾

ต้องแยกระหว่าง "ตัดสินใจ" กับ "ลงมือ":

Manager — ตัดสินใจ ✅
เลือก Renew / Cancel / Transfer ใน Power Apps
ไม่ต้องเข้า Vendor Portal
IT Admin — ลงมือ ✅
เข้า Vendor Portal / Billing Email บริษัท
Cancel หรือ Renew จริงๆ แล้ว mark Done ใน Power Apps

พนักงาน (License Owner) รับทราบเท่านั้น ❌ ไม่มี access Vendor Portal เพราะ License ผูกกับ Billing Email บริษัท ไม่ใช่ Email ส่วนตัว

SharePoint — Data Model
❓
ควรเก็บ ExpiryDate ที่ SoftwareCatalog หรือ Licenses List?
Design DecisionWhy
▾

ขึ้นอยู่กับโมเดล License ขององค์กร:

SoftwareCatalog.ExpiryDate
ใช้เมื่อเป็น Company Subscription
เช่น ซื้อ Adobe CC ทั้งองค์กร หมดพร้อมกันหมดทุกคน
→ Flow เช็คที่ Catalog ส่ง 1 Email ต่อ Software
Licenses.ExpiryDate
ใช้เมื่อเป็น Per-Seat License
เช่น พนักงาน A key หมด 15 มี.ค., B หมด 30 มิ.ย.
→ Flow เช็คทีละ License รายคน
📌 ในทางปฏิบัติองค์กรส่วนใหญ่มีทั้งสองแบบผสมกัน เช่น Microsoft 365 = subscription ทั้งองค์กร, Adobe = per-seat key
❓
4 SharePoint Lists แต่ละ List มีหน้าที่อะไร?
How It Works
▾

SoftwareCatalog Master data รายการซอฟต์แวร์ทั้งหมดที่บริษัทใช้ — Vendor, Category ไม่เปลี่ยนบ่อย

ApprovalRequests คำขอ License ทุกรายการ — มี PaymentMethod, PONumber/CCRefID, PurchaseStatus เป็น "ประตู" เปิด Phase 3

Licenses License ที่ assign ให้พนักงานจริงๆ — LicenseKey, AssignedTo, ExpiryDate, MovementStatus, ExpiryActionStatus

LicenseHistory Log ทุก action อัตโนมัติ — Flow เขียนให้ ใช้เพื่อ audit trail

❓
PurchaseStatus มี Choices อะไรบ้าง และสำคัญยังไง?
Critical Field
▾

Choices: Pending → In Process → Completed → Cancelled

Pending — สร้างตอน Submit Request (ยังรอ Approve)

In Process — Manager Approve แล้ว รอ Purchase ดำเนินการ

Completed — HR/IT Admin กรอก Write-Back เสร็จแล้ว ⚡ Trigger Flow 2 ทันที

Cancelled — Request ถูก Reject หรือยกเลิก

🔑 PurchaseStatus = "Completed" คือ trigger ที่ Flow 2 คอยดูอยู่ เมื่อเปลี่ยนเป็น Completed → Flow 2 ทำงาน → แจ้ง IT Admin ให้ Assign License

Power Automate Flows
❓
มี 4 Flows อะไรบ้าง แต่ละ Flow ทำอะไร?
How It Works
▾

Flow 1 — Request & Approval (PowerApps Trigger): รับ Request → บันทึก SharePoint → ส่ง Approval ให้ Manager → Approve/Reject → Route ตาม PaymentMethod

Flow 2 — Purchase Complete Trigger (SP Item Modified): คอยดู PurchaseStatus — เมื่อ = "Completed" → แจ้ง IT Admin ให้ Assign License + บันทึก LicenseHistory

Flow 3 — Expiry Action Flow (Recurrence ทุกวัน 08:00): ตรวจ License ที่ Active → 30 วัน/7 วัน/1 วัน/หมดแล้ว → บังคับให้มีคนรับผิดชอบจนถึง "Done"

Flow 4 — License Movement (PowerApps Trigger): รับ Action จาก IT Admin → Transfer/Return to Pool/Cancel → อัปเดต SharePoint + Email + LicenseHistory

❓
Flow 3 ต่างจาก "แค่แจ้งเตือนธรรมดา" ยังไง?
Priority 0Design Decision
▾

Flow 3 ไม่ใช่แค่ส่ง Email — มันบังคับให้มีคน ตัดสินใจและลงมือ จนกว่าจะ mark "Done":

1. Set ExpiryActionStatus = "Pending Decision" → แสดงขึ้นใน Power Apps ให้ Manager/IT Admin มาเลือก

2. ถ้าไม่มีใค Action ใน 7 วัน → Escalate ส่งซ้ำ

3. IT Admin ลงมือที่ Vendor Portal → กดปุ่ม "Done" ใน Power Apps → ExpiryActionStatus = "Done"

Pattern ที่ผิด: ส่ง Email แจ้งเตือน → จบ (ไม่รู้ว่าใครทำอะไรไปบ้าง)
Pattern ที่ถูก: แจ้ง → รอ Decision → รอ Action → ยืนยัน Done (มี audit trail ทุกขั้น)
❓
Flow 3 รันทุกวัน จะแจ้งเตือนซ้ำทุกวันไหม?
How It Works
▾

ไม่ เพราะ Filter Query ใน Step 1 มีเงื่อนไข: Status eq 'Active' and ExpiryActionStatus ne 'Done'

พอ IT Admin mark Done แล้ว → License นั้นจะไม่ถูก process อีก ส่วนการแจ้งเตือนซ้ำจะเกิดเฉพาะตาม threshold ที่กำหนด (30 วัน, 7 วัน, 1 วัน) ไม่ใช่ทุกวัน

Power Apps
❓
ทำไมไม่สร้าง Power Apps แยกสำหรับจัดซื้อและการเงิน?
ตัดออกWhy
▾

เพราะจัดซื้อและการเงินแค่ต้องกรอก Reference Number 1 field เท่านั้น การสร้าง App แยกทั้งหมดสำหรับ 1 field เป็น overkill และเสียเวลา 1–2 วัน

วิธีที่ดีกว่า: เพิ่ม Write-Back Section เล็กๆ ในหน้า Admin เดิม ให้ HR/IT Admin รับข้อมูลจากจัดซื้อ/การเงิน แล้วกรอกเอง

❓
3 Screens ใน Power Apps ทำอะไรบ้าง?
How It Works
▾

Screen 1 — Request Form (พนักงานใช้): กรอกชื่อ Software, จำนวน, เหตุผล, เลือก PaymentMethod → Submit → Trigger Flow 1

Screen 2 — Dashboard (ทุกคนดูได้): ภาพรวม Active Licenses, Expiry ใกล้หมด, Pending Purchases counter (รอ Write-Back)

Screen 3 — Admin (HR/IT Admin): Write-Back Section, Expiry Decision Panel, License Movement Panel, License CRUD

🧹
สิ่งที่ตัดออก และเหตุผล
❓
ทำไมไม่ทำ Power BI Dashboard ใน MVP?
ตัดออก
▾

เพราะ Power BI ต้องการ data ที่สะสมมาก่อนถึงจะมีประโยชน์ ถ้าระบบยังไม่ได้ใช้งาน data ก็ยังไม่มี

ลำดับความสำคัญ: ทำให้ระบบทำงานได้ก่อน → เก็บ data → ค่อยทำ BI ทีหลัง ทำ Power BI ตอนนี้คือ scope creep ที่ทำให้ไม่เสร็จทัน 31 มีนาคม

❓
ทำไมไม่ทำ Multi-Level Approval?
ตัดออก
▾

Multi-Level (Reviewer → Approver → หัวหน้า) ทำให้ Flow ซับซ้อนมาก, test ยาก, และถ้า Flow เสียหายจะหาสาเหตุได้ยากมาก

สำหรับ MVP ให้ Manager คนเดียว Approve ก่อน ถ้าองค์กรต้องการ Multi-Level จริงๆ ค่อย add ทีหลังหลัง Go-Live

❓
Flow 4 และ 5 เดิม (v1) คืออะไร ทำไมตัดออก?
ตัดออก
▾

Flow 4 เดิม = "Request Notification" (แยกออกมาส่ง Email แจ้งว่า Request ผ่าน)
Flow 5 เดิม = "Route Notification" (แยกออกมาบอกว่าไปทาง PR/PO หรือ Corporate Card)

ทั้งสองนี้ควรเป็น branch ใน Flow 1 อยู่แล้ว ไม่ควรแยก Flow ออกมา การแยกทำให้ debug ยาก dependency ซับซ้อน และไม่มีเหตุผลที่ดีพอ

Flow 4 ใหม่ = License Movement (Priority 0 ของจริง) แทนที่ของเดิม

❓
HTML Email Template สวยงาม ทำตอนไหน?
เลื่อนออกไป
▾

เลื่อนออกไปหลัง Go-Live เพราะ plain text Email ทำงานได้ครบฟังก์ชัน การออกแบบ HTML Email ใช้เวลาและไม่ใช่ blocker สำหรับการทำงานของระบบ

ทำ HTML Email ใน Sprint 2 หรือหลัง UAT ผ่านแล้ว
